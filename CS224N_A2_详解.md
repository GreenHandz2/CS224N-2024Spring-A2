## 1. 作业概述

### 1.1 我们要解决什么问题？

给定一句话 **"I eat apples"**，让计算机自动分析出：

```
         eat
        /   \
       I   apples
```

- `I` 是 `eat` 的**主语**（head = eat, dependent = I）
- `apples` 是 `eat` 的**宾语**（head = eat, dependent = apples）
- `eat` 是整个句子的**根**（head = ROOT, dependent = eat）

### 1.2 解决方案的设计思路

我们把问题拆成两部分：

```
┌──────────────────────────────────────────────┐
│  规则系统（parser_transitions.py）            │
│  → 定义了三种操作（S/LA/RA）来构建依存树      │
│  → 但它不知道每一步该选哪个操作               │
└───────────────────┬──────────────────────────┘
                    │ 每一步问："该做什么？"
                    ▼
┌──────────────────────────────────────────────┐
│  神经网络（parser_model.py）                 │
│  → 看当前状态，输出 3 个分数（S/LA/RA）       │
│  → 取最高分的作为决策                        │
└───────────────────┬──────────────────────────┘
                    │ 需要训练
                    ▼
┌──────────────────────────────────────────────┐
│  训练系统（run.py）                           │
│  → 用标准答案训练网络，让它学会做正确的决策    │
│  → 前向传播 → 算loss → 反向传播 → 更新参数    │
└──────────────────────────────────────────────┘
```

### 1.3 评估指标：UAS

**UAS（Unlabeled Attachment Score）= 预测对 head 的词数 / 总词数**

- 只看 head 对不对，不管关系类型（主语/宾语等标签）
- 例如："I eat apples" 中有 3 个词，2 个 head 猜对了，UAS = 66.7%
- 70%+ 算及格，80%+ 算优秀

## 2. 背景知识：依存句法分析

### 2.1 Head（中心词）和 Dependent（依存词）

**直觉类比：家族树**

```
        祖父
        /   \
     爸爸    叔叔
     /  \
   我   妹妹
```

- "我" 的爸爸是谁？→ "爸爸"
- "爸爸" 的爸爸是谁？→ "祖父"

在句法中同理：

```
        eat（动词=句子的根）
       /   \
     I     apples
   （主语）  （宾语）
```

| Head（被依赖者） | Dependent（依赖者） | 关系解释 |
|:---:|:---:|------|
| eat | I | "I" 是 "eat" 的主语 |
| eat | apples | "apples" 是 "eat" 的宾语 |
| ROOT | eat | "eat" 是句子核心 |

**核心规则**：在每一对依存关系中，**Head 是"更重要的那个"，Dependent 是"依附于别人的那个"**。

> 比如"the cat"中，cat 比 the 更重要（cat 是名词核心，the 只是修饰），所以 cat 是 head，the 是 dependent。

### 2.2 数据格式：CoNLL

训练数据是 CoNLL 格式的文本文件（[data/train.conll](data/train.conll)），每行一个词，共 10 列：

```
索引  词形  _  POS标签  POS细类  _  HEAD索引  依存关系标签  _  _
1     In    _  ADP      IN      _  5          case          _  _
2     an    _  DET      DT      _  5          det           _  _
5     review _ NOUN     NN      _  45         nmod          _  _
```

- 第 7 列（HEAD 索引）：这个词的 head 是哪个词（索引）
- 第 8 列（依存关系标签）：head 和 dependent 之间是什么关系
- 索引为 0 代表 ROOT（句子的根）

### 2.3 为什么选择 Transition-Based 方法？

Transition-Based 解析是 **O(n)** 线性时间的，而基于图的解析是 **O(n³)** 的。简单来说——**快**。Chen & Manning (2014) 证明简单的神经网络 + 转换系统可以达到很高的准确率。

## 3. Arc-Standard 转换系统

### 3.1 三个数据结构

```
┌─────────────┐    ┌──────────────────┐    ┌────────────────┐
│   STACK     │    │     BUFFER       │    │  DEPENDENCIES  │
│   (栈)      │    │   (缓冲区)       │    │  (依存关系)    │
│   顶端在末尾 │    │   队首在开头     │    │  (head, dep)   │
├─────────────┤    ├──────────────────┤    ├────────────────┤
│   "ROOT"    │    │  "the"           │    │                │
│             │    │  "cat"           │    │                │
│             │    │  "sat"           │    │                │
└─────────────┘    └──────────────────┘    └────────────────┘
```

- **Stack（栈）**：存放正在处理的词，后进先出（LIFO）。初始只含 `"ROOT"`
- **Buffer（缓冲区）**：存放还没处理的词，先进先出（FIFO）。初始包含句中所有词
- **Dependencies（依存关系）**：已发现的 `(head, dependent)` 元组列表

### 3.2 三种操作

#### SHIFT（推进）— "S"

```
操作：buffer.pop(0) → stack.append()
含义：从锅里捞一个菜到左手筷子上

操作前：stack=["ROOT"]         buffer=["the","cat","sat"]
操作后：stack=["ROOT","the"]   buffer=["cat","sat"]
```

#### LEFT-ARC（左弧）— "LA"

```
操作：建立 stack[-1] → stack[-2] 的依存关系，移除 stack[-2]
含义：栈顶是 head，它往左指着倒数第二个词

记忆口诀：Left-Arc = ←────── 箭头向左

操作前：stack=["ROOT","the","cat"]  buffer=["sat"]     deps=[]
         head = stack[-1] = "cat"
         dep  = stack[-2] = "the"
         
操作后：stack=["ROOT","cat"]   buffer=["sat"]   deps=[("cat","the")]
         ↑ "the" 被移除了，"cat" 是 "the" 的 head
```

**为什么 LA 移除的是 stack[-2] 而不是 stack[-1]？**

因为 head（stack[-1]="cat"）要留在栈上，继续和栈里其他词建立关系。被移除的 dependent（stack[-2]="the"）已经找到了 head，它的使命完成了。

#### RIGHT-ARC（右弧）— "RA"

```
操作：建立 stack[-2] → stack[-1] 的依存关系，移除 stack[-1]
含义：倒数第二个是 head，它往右指着栈顶的词

记忆口诀：Right-Arc = ──────→ 箭头向右

操作前：stack=["ROOT","cat","sat"]  buffer=[]     deps=[("cat","the")]
         head = stack[-2] = "cat"
         dep  = stack[-1] = "sat"
         
操作后：stack=["ROOT","cat"]   buffer=[]   deps=[("cat","the"),("cat","sat")]
         ↑ "sat" 被移除了，"cat" 是 "sat" 的 head
```

**为什么 RA 移除的是 stack[-1] 而不是 stack[-2]？**

同样，head（stack[-2]="cat"）留在栈上继续工作，dependent（stack[-1]="sat"）因为找到了 head 而被移走。

### 3.3 LA vs RA 对比表

| 对比维度 | LEFT-ARC (LA) | RIGHT-ARC (RA) |
|:---:|:---:|:---:|
| 箭头方向 | ← 向左 | → 向右 |
| Head 位置 | stack[-1]（栈顶） | stack[-2]（倒数第二） |
| Dependent 位置 | stack[-2]（倒数第二） | stack[-1]（栈顶） |
| 移除谁？ | stack[-2]（左边的） | stack[-1]（右边的） |
| 谁留在栈上？ | head（栈顶） | head（倒数第二） |

**速度记忆法**：LA 的 "L" 对应 Left（左），所以移除的是左边那个（stack[-2]）。RA 的 "R" 对应 Right（右），所以移除的是栈顶那个（右边那个，stack[-1]）。

### 3.4 完整示例演示："the cat sat"

```
句子: "the cat sat"

初始: stack=["ROOT"]              buffer=["the","cat","sat"]   deps=[]

① S    stack=["ROOT","the"]       buffer=["cat","sat"]        deps=[]
② S    stack=["ROOT","the","cat"] buffer=["sat"]              deps=[]
③ LA   stack=["ROOT","cat"]       buffer=["sat"]              deps=[("cat","the")]
       ↑ "cat" 是 "the" 的 head（"the cat" 中 cat 更重要）

④ S    stack=["ROOT","cat","sat"] buffer=[]                   deps=[("cat","the")]
⑤ RA   stack=["ROOT","cat"]       buffer=[]                   deps=[("cat","the"),("cat","sat")]
       ↑ "cat" 是 "sat" 的 head（"cat sat" 中 cat 是主语，sat 是谓语）

⑥ RA   stack=["ROOT"]             buffer=[]                   deps=[("cat","the"),("cat","sat"),("ROOT","cat")]
       ↑ "ROOT" 是 "cat" 的 head（cat 是整个句子的根词）

完成！依存树：
         ROOT
          |
         cat
        /   \
      the   sat
```

### 3.5 终止条件

当 **buffer 为空** 且 **stack 只剩 `["ROOT"]`** 时，解析完成：

```python
len(p.buffer) == 0 and len(p.stack) == 1  # stack中只剩ROOT
```

每个词都被移入了 stack 然后又建立了依存关系被移走，最终所有词都通过一条弧连接到了 ROOT。

## 4. parser_transitions.py 详解

### 4.1 文件总览

这个文件实现了**解析引擎**——它不决定"该做什么"，只负责"执行操作"。

| 类/函数 | 作用 | 对应作业部分 |
|---------|------|:---:|
| `PartialParse.__init__` | 初始化 stack/buffer/dependencies | Part C |
| `PartialParse.parse_step` | 执行一步 S/LA/RA 操作 | Part C |
| `PartialParse.parse` | 连续执行一系列操作 | — |
| `minibatch_parse` | 批量并行解析多个句子 | Part D |

### 4.2 `PartialParse.__init__` — 初始化解析状态

```python
def __init__(self, sentence):
    self.sentence = sentence         # 原始句子，不动它
    self.stack = ["ROOT"]            # 栈初始只有根节点
    self.buffer = list(sentence)     # buffer是句子的拷贝
    self.dependencies = []           # 还没有发现依存关系
```

#### 代码要点

**① 为什么 `self.stack = ["ROOT"]`？**

ROOT 是人造的特殊节点，代表"句子的根"。每个句子最终都有一条从 ROOT 指向根词的弧。如果把依存树想象成一棵树，ROOT 就是树根之上的"地基"。没有 ROOT，根词就找不到 head，解析永远不能终止。

**② 为什么 `list(sentence)` 而不是直接 `sentence`？**

```python
# 错误做法：
self.buffer = sentence   # buffer 和 sentence 指向同一个列表对象

# 正确做法：
self.buffer = list(sentence)  # 创建一个新列表，内容相同但互不影响
```

Python 中 `=` 传递的是**引用**（指向同一个内存地址）。如果 `self.buffer = sentence`，后续对 buffer 的 `pop(0)` 等修改操作也会改变原句。注释明确说"Do NOT modify the sentence object"。

`list()` 构造器创建了一个**浅拷贝**——新列表对象，但里面的元素（单词字符串）引用相同。字符串本身是不可变的，所以浅拷贝完全安全。

**③ 为什么 `self.sentence = sentence` 不用拷贝？**

sentence 只被用于"句子身份标识"（判断两个 PartialParse 是否属于同一个句子），不会对它做修改操作，直接引用即可。

### 4.3 `PartialParse.parse_step` — 执行一步操作

```python
def parse_step(self, transition):
    if transition == "S":
        # Shift: 把buffer的第一个词移到stack顶部
        self.stack.append(self.buffer.pop(0))

    elif transition == "LA":
        # Left-Arc: 栈顶(stack[-1])是head, 倒数第二(stack[-2])是dependent
        self.dependencies.append((self.stack[-1], self.stack[-2]))
        self.stack.pop(-2)

    elif transition == "RA":
        # Right-Arc: 倒数第二(stack[-2])是head, 栈顶(stack[-1])是dependent
        self.dependencies.append((self.stack[-2], self.stack[-1]))
        self.stack.pop(-1)
```

#### 代码要点

**① Shift：为什么是 `pop(0)` 和 `append`？**

```
Buffer 是队列（FIFO，先进先出）：队首在开头 → pop(0) 弹出第一个
Stack 是栈（LIFO，后进先出）：栈顶在末尾 → append() 追加到末尾
```

- `list.pop(0)` 弹出索引 0 的元素（队首），时间复杂度 O(n)（后续元素要前移）
- `list.append(x)` 追加到末尾（栈顶），时间复杂度 O(1)

**② Left-Arc：为什么是 `(self.stack[-1], self.stack[-2])` 这个顺序？**

```python
self.dependencies.append((self.stack[-1], self.stack[-2]))
#                          head ↑              dep ↑
```

元组格式是 `(head, dependent)`——head 在前，dependent 在后。LA 时，栈顶 [-1] 是 head，倒数第二 [-2] 是 dependent。所以写 `(stack[-1], stack[-2])`。

**③ Left-Arc：为什么 `pop(-2)`？**

```python
self.stack.pop(-2)   # 移除被 LA 建立关系的 dependent
```

`pop(-2)` 移除倒数第二个元素。dependent 找到了它的 head，从栈上移除。而 head（栈顶）留在栈上，因为它可能还要和栈里其他词建立关系。

**④ Right-Arc：为什么和 LA 反过来？**

```python
self.dependencies.append((self.stack[-2], self.stack[-1]))
#                          head ↑              dep ↑
self.stack.pop(-1)  # 移除栈顶的 dependent
```

RA 时，head 是倒数第二，dependent 是栈顶。所以内容上是 `(stack[-2], stack[-1])`，移除的是 `stack[-1]`。

**⑤ 为什么 S 操作不需要条件判断？**

LA 和 RA 都要求栈中至少有两个元素（一个 head + 一个 dependent），而 S 只要 buffer 不空就能执行。代码假设了"传入的 transition 都是合法的"，所以不需要做防卫性判断。

### 4.4 `minibatch_parse` — 批量解析

```python
def minibatch_parse(sentences, model, batch_size):
    # ① 为每个句子创建一个 PartialParse 对象
    partial_parses = [PartialParse(s) for s in sentences]
    unfinished_parses = partial_parses[:]   # 浅拷贝
    
    # ② 只要还有句子没解析完，就继续循环
    while unfinished_parses:
        # 取 batch_size 个为一组
        batch = unfinished_parses[:batch_size]
        # 让模型预测这组每个句子下一步该做什么
        transitions = model.predict(batch)
        # 逐个执行模型预测的操作
        for parse, transition in zip(batch, transitions):
            parse.parse_step(transition)
        # 过滤出还没完成的句子
        unfinished_parses = [p for p in unfinished_parses
                             if not (len(p.buffer) == 0 and len(p.stack) == 1)]
    
    # ③ 返回所有句子的依存关系
    dependencies = [pp.dependencies for pp in partial_parses]
    return dependencies
```

#### 代码要点

**① 为什么需要 `partial_parses` 和 `unfinished_parses` 两个列表？**

```python
partial_parses = [PartialParse(s) for s in sentences]   # 原始列表
unfinished_parses = partial_parses[:]                     # 工作列表
```

- `partial_parses`：保持不动，最后要从中取 `pp.dependencies` 作为返回结果
- `unfinished_parses`：会不断被过滤缩小，用于判断循环终止

如果你直接对 `partial_parses` 做过滤操作，最后就拿不到已经完成的句子的解析结果了。

**② 为什么用浅拷贝 `[:]` 而不是深拷贝？**

```python
unfinished_parses = partial_parses[:]
```

浅拷贝创建一个**新的列表对象**，但里面的元素（PartialParse 实例）是**同一批对象**。意味着：

```python
# 通过 batch 中任一列表修改 parse 的状态
parse.parse_step(transition)

# 这个修改同时反映在 partial_parses 和 unfinished_parses 中
# 因为它们指向的是同一个 PartialParse 对象
```

这正是我们需要的！如果你做了深拷贝（`copy.deepcopy`），修改 batch 里的 parse 不会影响 `partial_parses`，最终取出来的 dependencies 全是空的。

**③ 为什么不能使用 `del`？**

注释里的原话：

> you should NOT use the `del` operator to remove objects from the `unfinished_parses` list. This will free the underlying memory that is being accessed by `partial_parses` and may cause your code to crash.

```python
# 错误做法：
del unfinished_parses[i]   # 可能释放 PartialParse 对象的内存
                           # 但 partial_parses 还在引用它 → 野指针 → 崩溃

# 正确做法：
unfinished_parses = [p for p in unfinished_parses if condition]
# 列表推导式创建了一个新的引用列表
# 不满足条件的对象只是不再被 unfinished_parses 引用
# 但它们仍然被 partial_parses 引用，所以不会被垃圾回收
```

**④ 终止条件为什么是这两个？**

```python
len(p.buffer) == 0    # 所有词都从 buffer 移走了
and len(p.stack) == 1 # 栈上只剩下 ROOT
```

当句子中的所有词都经由 stack 被处理、被分配了 head 并移除后，栈上最终只剩 ROOT。如果一个句子成功解析完成，这个条件必定为真。

**⑤ 为什么是 `batch = unfinished_parses[:batch_size]`？**

取前 `batch_size` 个。如果剩余不足 `batch_size`，全部取走。这是 Python 切片的特性——不会越界：

```python
[1,2,3][:5]  # 返回 [1,2,3]，不报错
```

## 5. parser_model.py 详解

### 5.1 文件总览

这个文件实现了**神经网络模型**——给定当前解析状态的 36 个特征（词 ID），输出 3 个类别的分数（S/LA/RA）。

| 方法 | 作用 | 对应作业部分 |
|------|------|:---:|
| `__init__` | 定义网络参数：权重、偏置、Dropout | Part E |
| `embedding_lookup` | 把词 ID 转换为词向量 | Part F |
| `forward` | 实现完整的前向传播 | Part G |

### 5.2 网络架构全景图

```
输入层                      隐藏层                       输出层
36个特征ID  ─────────→   200个神经元   ─────────→    3个类别
                         ReLU激活                     (S, LA, RA)
                         Dropout(0.5)
                         
(batch,36)               (batch,200)                 (batch,3)
    │                        │                          │
    │  embedding_lookup       │  ×W1+b1+ReLU+Drop       │  ×W2+b2
    │  每个ID→50维向量        │                          │
    │  36×50=1800拼接         │                          │
    ▼                        ▼                          ▼
(batch,1800) ───────────→ (batch,200) ────────────→ (batch,3)
         ×W1(1800,200)               ×W2(200,3)
         +b1(200,)                   +b2(3,)
```

#### 形状变换总结

```
w                  (batch_size, 36)         ← 36个特征ID
    ↓ embedding_lookup
嵌入后              (batch_size, 1800)       ← 36×50=1800
    ↓ matmul(W1) + b1
隐藏层              (batch_size, 200)        ← 200个隐藏单元
    ↓ ReLU
激活后              (batch_size, 200)        ← 负数已清零
    ↓ Dropout
防过拟合后          (batch_size, 200)        ← ~50%神经元为0
    ↓ matmul(W2) + b2
logits              (batch_size, 3)          ← [S分, LA分, RA分]
```

### 5.3 `__init__` — 定义网络参数（Part E）

```python
def __init__(self, embeddings, n_features=36,
    hidden_size=200, n_classes=3, dropout_prob=0.5):
    
    super(ParserModel, self).__init__()
    self.n_features = n_features
    self.n_classes = n_classes
    self.dropout_prob = dropout_prob
    self.embed_size = embeddings.shape[1]      # 词向量维度=50
    self.hidden_size = hidden_size
    self.embeddings = nn.Parameter(torch.tensor(embeddings))

    # ① 输入→隐藏层的权重和偏置
    self.embed_to_hidden_weight = nn.Parameter(
        torch.empty(self.n_features * self.embed_size, self.hidden_size)
    )
    nn.init.xavier_uniform_(self.embed_to_hidden_weight)
    
    self.embed_to_hidden_bias = nn.Parameter(
        torch.empty(self.hidden_size)
    )
    nn.init.uniform_(self.embed_to_hidden_bias)

    # ② Dropout层
    self.dropout = nn.Dropout(self.dropout_prob)

    # ③ 隐藏→输出的权重和偏置
    self.hidden_to_logits_weight = nn.Parameter(
        torch.empty(self.hidden_size, self.n_classes)
    )
    nn.init.xavier_uniform_(self.hidden_to_logits_weight)
    
    self.hidden_to_logits_bias = nn.Parameter(
        torch.empty(self.n_classes)
    )
    nn.init.uniform_(self.hidden_to_logits_bias)
```

#### 代码要点

**① 为什么用 `nn.Parameter` 而不是 `nn.Linear`？**

```python
# 这个作业的做法（手写）：
self.embed_to_hidden_weight = nn.Parameter(torch.empty(1800, 200))
nn.init.xavier_uniform_(self.embed_to_hidden_weight)

# PyTorch 内置的等价做法：
self.linear = nn.Linear(1800, 200)  # 自动创建weight和bias，自动Xavier初始化
```

作业要求手写是为了让你**理解 PyTorch 的底层机制**：

- `nn.Parameter` 是 `torch.Tensor` 的子类，但多了一个特殊标记：它会被自动注册为模型的"可训练参数"
- 只有 `nn.Parameter` 包裹的张量才会被 `model.parameters()` 返回
- 只有 `model.parameters()` 返回的参数才会被优化器追踪和更新
- 反向传播时，PyTorch autograd 只为 `nn.Parameter` 计算梯度

```python
# 验证：非 Parameter 的张量不会被训练
x = torch.tensor([1.0])              # 普通张量
p = nn.Parameter(torch.tensor([1.0])) # Parameter

model.parameters()  # 只包含 p，不包含 x
```

**② 权重形状为什么是 `(1800, 200)` 和 `(200, 3)`？**

这是矩阵乘法的维度约束决定的：

```python
# 前向传播中的矩阵乘法
X @ W1 = (batch, 1800) @ (1800, 200) = (batch, 200)
#         ↑ 列数必须等于    ↑ 行数

H @ W2 = (batch, 200) @ (200, 3) = (batch, 3)
#         ↑ 列数必须等于  ↑ 行数
```

矩阵乘法的铁律：**A 的列数必须等于 B 的行数**。所以 W1 的第一维是 1800（输入维度），W2 的第一维是 200（隐藏层维度）。

**③ Xavier 初始化的原理是什么？**

```python
nn.init.xavier_uniform_(self.embed_to_hidden_weight)
```

Xavier（Glorot）初始化的核心思想：**让每一层输入和输出的方差保持一致**，防止信号在深层网络中爆炸或消失。

对于一个形状为 `(n_in, n_out)` 的权重矩阵：

$$W \sim U\left(-\frac{\sqrt{6}}{\sqrt{n_{in} + n_{out}}}, \frac{\sqrt{6}}{\sqrt{n_{in} + n_{out}}}\right)$$

对于 `(1800, 200)` 的矩阵：

$$a = \frac{\sqrt{6}}{\sqrt{1800 + 200}} = \frac{2.449}{\sqrt{2000}} = \frac{2.449}{44.72} \approx 0.0548$$

$$W \sim U(-0.0548, 0.0548)$$

对比普通随机初始化（比如 `U(-1, 1)`），Xavier 把值压缩了约 20 倍，避免了信号在网络中指数级放大。

**为什么偏置用 Uniform 而不用 Xavier？**

偏置只是一个小的平移量，对信号方差的影响远小于权重矩阵。用简单的均匀随机数初始化即可。`nn.init.uniform_` 默认范围是 (0, 1)。

**④ 为什么 `torch.empty()` 然后 `init`，而不是一步完成？**

```python
# 两步走：
w = torch.empty(1800, 200)       # 分配内存（内容是垃圾值）
nn.init.xavier_uniform_(w)       # 原地覆盖为Xavier分布的值

# 不能一步完成的原因：
# nn.init 函数都是 IN-PLACE 操作（函数末尾有 _）
# 它们需要已存在的张量来写入
```

带 `_` 后缀的 PyTorch 函数是 in-place 操作，就像 `list.sort()` 和 `list.append()` 一样直接修改原对象。

**⑤ Dropout 的概率为什么是 0.5？**

```python
self.dropout = nn.Dropout(0.5)   # 训练时随机丢弃50%的神经元
```

- 0.5 是隐藏层的经验最佳值（Srivastava et al., 2014）
- 太低（如 0.1）：基本没效果，网络容易过拟合
- 太高（如 0.9）：太多神经元被关，网络学不到东西
- 输入层通常用更低的 dropout（如 0.2），但这个作业没在 embedding 层用 dropout

### 5.4 深度解析：Dropout 原理

#### Dropout 解决什么问题？

神经网络的一个致命弱点：**过拟合（Overfitting）**——模型在训练数据上表现完美，但在新数据上表现糟糕。原因是神经元之间出现了**共适应（Co-adaptation）**：某些神经元过度依赖另一些神经元的存在，形成了"作弊小团体"。

#### Dropout 怎么解决？

```
训练时（每轮随机关闭不同的神经元）：
  输入 → [● ✕ ● ✕ ● ✕] → 输出    （取平均值）
  
测试时（所有神经元都工作，但输出要缩放）：
  输入 → [● ● ● ● ● ●] → 输出×0.5  （等价于训练时的期望）
```

想象一个有 6 个学生的学习小组：
- 没有 Dropout：每次讨论，所有人都发言。两个学霸主导了讨论，其他 4 人跟着附和。考试时，学霸请假了→ 小组完全答不出来。
- 有 Dropout：每次讨论，随机抽掉 3 个人（不能发言）。学霸不可能每次都发言，其他人被迫独立思考。考试时，不管谁请假，剩下的人都能独立应对。

**PyTorch 里的具体实现**：

```python
# 训练时（model.train()）：
self.dropout = nn.Dropout(0.5)
# 生成一个 (0,1) 的随机掩码
# mask > 0.5 的神经元保留，< 0.5 的归零
# 保留的神经元值 ×2（除以保留概率1-p）

# 推理时（model.eval()）：
# Dropout 不生效，所有值原样通过
```

**为什么 assessment 时需要 `model.eval()`？**

评估时你要用的是模型的最佳表现（所有神经元全力输出），而不是带噪声的表现。`model.eval()` 自动关闭 Dropout。

### 5.5 `embedding_lookup` — 查字典（Part F）

```python
def embedding_lookup(self, w):
    x = self.embeddings[w]         # (batch_size, n_features, embed_size)
    x = x.view(x.shape[0], -1)     # (batch_size, n_features * embed_size)
    return x
```

#### 代码要点

**① `self.embeddings[w]` 发生了什么？**

`self.embeddings` 是一个 `(词表大小, 50)` 的大矩阵：

```python
# 类似于：
embeddings = np.array([
    [0.12, -0.34, 0.56, ..., 0.78],   # 词ID=0  → "the" 的50维向量
    [0.45, 0.21, -0.67, ..., 0.33],  # 词ID=1  → "cat" 的50维向量
    [-0.23, 0.89, 0.12, ..., -0.44], # 词ID=2  → "sat" 的50维向量
    ...
])  # shape: (词表大小, 50)
```

`w` 是一个 `(batch_size, 36)` 的整数矩阵：

```python
w = [[17, 3, 88, 5, ...],   # 第1个样本的36个特征ID
     [5,  17, 2, 42, ...],   # 第2个样本的36个特征ID
     ...]
```

PyTorch 的**高级索引（Advanced Indexing）**：当你用一个整数张量作为索引时，PyTorch 会对该张量中的每个整数，去源张量中取出对应的行。

```python
self.embeddings[w]  # w中每个整数i → 取出embeddings的第i行

# 伪代码展示这个过程：
result = []
for sample in w:              # 遍历每个样本
    sample_vectors = []
    for word_id in sample:     # 遍历36个特征ID
        sample_vectors.append(embeddings[word_id])  # 取出50维向量
    result.append(sample_vectors)

# result形状：(batch_size, 36, 50)
```

**② 为什么不能用 `nn.Embedding`？**

这是作业的考点——让你亲手实现 Embedding 层：

```python
# 你的实现（手写）：
x = self.embeddings[w]       # 索引查找
x = x.view(x.shape[0], -1)   # 拍扁

# PyTorch内置（原理完全一样）：
embed = nn.Embedding(num_embeddings, embedding_dim)
x = embed(w)                  # 内部就是索引查找
```

`nn.Embedding` 的源码本质上就是存了一个 `(词表大小, 嵌入维度)` 的权重矩阵，然后做索引查找。现在你知道了它的底层实现。

**③ `view(x.shape[0], -1)` 的作用是什么？**

```python
x.shape  # (4, 36, 50)  ← batch=4, 36个特征, 每个50维

x.view(x.shape[0], -1)  # (4, 1800)  ← batch=4, 36×50=1800拼成一排
```

- `x.shape[0]`：保留 batch 维度不变（= 4）
- `-1`：让 PyTorch 自动计算"剩下需要多少"（36×50 = 1800）

**为什么要拍扁？** 因为下一步是矩阵乘法 `X @ W1`。矩阵乘法要求输入是 2D（矩阵），不能是 3D。把 36 个 50 维向量拼接成 1800 维后，每个样本就变成了一个一维向量，整个 batch 就是一个 `(batch, 1800)` 的矩阵。

**④ 怎么理解词向量（Word Embedding）？**

词向量就是"用数字表示一个词"。每个词被表示为一个 50 维的实数向量：

```
"cat"  → [0.23, -0.45, 0.67, ..., 0.12]  (50个数字)
"dog"  → [0.21, -0.43, 0.65, ..., 0.15]  (和cat的向量很接近!)
"apple"→ [0.78, 0.32, -0.21, ..., -0.55] (和cat的向量较远)
```

通过训练，语义相近的词在向量空间中距离更近。`cosine("cat", "dog")` 比 `cosine("cat", "apple")` 大得多。这些向量是预先在大规模语料（如维基百科）上训练好的（本文使用 GloVe），我们在解析器训练时可以直接复用。

# 中间有一条空行

### 5.6 `forward` — 前向传播（Part G）

```python
def forward(self, w):
    x = self.embedding_lookup(w)                          # (batch, 1800)
    x = torch.matmul(x, self.embed_to_hidden_weight)      # (batch, 200)
    x = x + self.embed_to_hidden_bias                     # (batch, 200)
    x = F.relu(x)                                         # (batch, 200)
    x = self.dropout(x)                                   # (batch, 200)
    x = torch.matmul(x, self.hidden_to_logits_weight)     # (batch, 3)
    x = x + self.hidden_to_logits_bias                    # (batch, 3)
    logits = x                                            # (batch, 3)
    return logits
```

#### 代码要点

**① 逐行解析**

```python
# 第1行：查字典取词向量
x = self.embedding_lookup(w)     # (4, 36) → (4, 1800)

# 第2行：矩阵乘法：输入→隐藏层
x = torch.matmul(x, self.embed_to_hidden_weight)
# (4, 1800) @ (1800, 200) = (4, 200)
# 每个样本的1800维特征被映射到了200维的隐藏空间

# 第3行：加偏置
x = x + self.embed_to_hidden_bias  # (4, 200) + (200,) = (4, 200)
# PyTorch广播机制：自动把(200,)扩展成(4,200)，逐元素相加

# 第4行：ReLU激活
x = F.relu(x)                     # (4, 200)
# 每个元素：max(0, x)

# 第5行：Dropout正则化
x = self.dropout(x)               # (4, 200)
# 训练时：约100个神经元被随机归零
# 评估时：什么都不做

# 第6行：矩阵乘法：隐藏→输出
x = torch.matmul(x, self.hidden_to_logits_weight)
# (4, 200) @ (200, 3) = (4, 3)
# 200维压缩到3维，每个样本得到3个分数

# 第7行：加偏置
x = x + self.hidden_to_logits_bias  # (4, 3) + (3,) = (4, 3)

# 第8行：赋值给logits
logits = x                         # (4, 3)
# logits[0] = [0.8, 0.3, 0.1]  → 模型认为第1个样本该做S(分数0.8最高)
```

**② 为什么用 `torch.matmul` 而不是 `@` 运算符？**

```python
# 两种写法完全等价：
x @ self.embed_to_hidden_weight         # Python 3.5+ 的 @ 运算符
torch.matmul(x, self.embed_to_hidden_weight)  # PyTorch 函数
```

作业文档推荐 `torch.matmul` 是为了清晰——让初学者明确这是矩阵乘法。两者底层调用的是一样的 CUDA 实现。

**③ 为什么要加偏置？**

如果没有偏置 `b`，模型只能表示"过原点"的线性函数：

```
y = Wx     ← 当 x=0 时，y 恒为 0（不够灵活）
y = Wx + b ← 当 x=0 时，y = b（可以任意偏移）
```

偏置给了每个神经元一个"基本兴奋度"。想象：没有偏置的神经元就像没装弹簧的门——推不动的永远推不动。有偏置就像装了弹簧——有一个"默认开度"。

**④ `F.relu(x)` 存在的意义是什么？**

ReLU = Rectified Linear Unit = 修正线性单元：

$$ReLU(x) = \max(0, x)$$

```
输入:  [0.3, -0.5, 0.8, -1.2, 0.0]
输出:  [0.3,  0.0, 0.8,  0.0, 0.0]
```

**如果没有 ReLU（或其他非线性函数），多层的线性变换等价于一层**：

```
X → W1 → W2 → W3

等价于：X → W_equivalent

因为：
f(x) = W2(W1(x)) = W2×W1×x = W_equivalent×x  （两个线性变换的复合还是线性变换）
```

这就像用三把尺子量东西，和用一把尺子量——本质上没有区别，只是乘了一个不同的系数。

**加入 ReLU 后**，网络的表达能力指数级提升——它可以近似任意连续函数（Universal Approximation Theorem）。

**⑤ 为什么偏置加法用 `x + b` 而不是 `torch.add`？**

PyTorch 支持**广播（Broadcasting）**：形状不同的张量在运算时，PyTorch 自动将小的扩展到和大的匹配。

```python
x.shape  # (4, 200)    ← batch=4，每行200维
b.shape  # (200,)       ← 只是一维向量

x + b    # 自动把 (200,) 广播成 (4, 200)
         # 相当于把 b 复制了4份，每份叠加到对应行
```

广播规则：从最后维度开始对齐，维度为 1 或缺失的维度被"拉伸"去匹配。这里是 `(4,200) + (200,)` → `(200,)` 被视为 `(1,200)` → 拉伸为 `(4,200)`。

**⑥ 为什么 forward 里不加 softmax？**

```python
# 你在 forward 里没有写：
# logits = F.softmax(x, dim=1)  ← 不写！

# 因为 nn.CrossEntropyLoss 已经包含了：
# loss = -log(softmax(logits)[true_class])
```

如果 forward 里加了 softmax，loss 函数里又来一次 softmax：

```
第一次softmax: [0.8, 0.3, 0.1] → [0.53, 0.32, 0.15]（变成概率）
第二次softmax: [0.53, 0.32, 0.15] → [0.38, 0.31, 0.31]（趋于均匀分布）
```

两次 softmax 会让输出趋于均匀分布，模型完全学不到东西。

`CrossEntropyLoss` 内部用 **LogSoftmax**（数值更稳定）计算：

$$loss = -\log\left(\frac{e^{logit_{true}}}{\sum_j e^{logit_j}}\right)$$

直接算 `e^x` 可能溢出，PyTorch 内部做了 log-sum-exp trick：

$$loss = -logit_{true} + \log\left(\sum_j e^{logit_j - \max(logit)}\right)$$

**⑦ `self.dropout(x)` 在 `model.eval()` 时自动失效吗？**

是的！`nn.Dropout` 内部检查 `self.training` 标志：

```python
# 训练时
model.train()           # 设置 self.training = True
x = self.dropout(x)     # Dropout 生效：随机丢弃神经元

# 评估时
model.eval()            # 设置 self.training = False
x = self.dropout(x)     # Dropout 不生效：原样输出
```

这就是为什么在 `train_for_epoch()` 里必须先调 `parser.model.train()`。

### 6.2 `train` — 训练总控（Part H）

```python
def train(parser, train_data, dev_data, output_path, 
          batch_size=1024, n_epochs=10, lr=0.0005):
    
    best_dev_UAS = 0

    # Part H: 创建优化器和损失函数
    optimizer = optim.Adam(parser.model.parameters(), lr=lr)
    loss_func = nn.CrossEntropyLoss()

    for epoch in range(n_epochs):
        print("Epoch {:} out of {:}".format(epoch + 1, n_epochs))
        dev_UAS = train_for_epoch(parser, train_data, dev_data, 
                                   optimizer, loss_func, batch_size)
        if dev_UAS > best_dev_UAS:
            best_dev_UAS = dev_UAS
            print("New best dev UAS! Saving model.")
            torch.save(parser.model.state_dict(), output_path)
```

#### 代码要点

**① 为什么选择 Adam 优化器？**

Adam（Adaptive Moment Estimation）是目前最流行的深度学习优化器之一，它结合了两种思想：

1. **Momentum（动量）**：像滚雪球——梯度一直朝一个方向时加速；方向改变时减速
2. **RMSprop**：自适应学习率——梯度大的参数减小学习率；梯度小的参数增大学习率

```python
# Adam的核心更新公式（简化版）：
m = β1 × m + (1-β1) × grad         # 一阶矩（梯度方向的历史平均值）
v = β2 × v + (1-β2) × grad²        # 二阶矩（梯度大小的历史平均值）

m̂ = m / (1-β1^t)                   # 偏差修正
v̂ = v / (1-β2^t)

param = param - lr × m̂ / (√v̂ + ε)  # 更新参数
```

对比普通 SGD：

| 特性 | SGD | Adam |
|------|-----|------|
| 学习率 | 固定 | 每个参数自适应 |
| 收敛速度 | 慢 | 快 |
| 调参难度 | 需要仔细调lr | 默认lr=0.001通常就行 |
| 适合场景 | 计算机视觉（需要精细调参） | NLP（词向量梯度差异大） |

**`lr=0.0005`**：这个作业用了一个比默认 0.001 小的学习率，因为词向量是预训练的，不宜大幅改动。

**② `parser.model.parameters()` 返回了什么？**

```python
parser.model = ParserModel(embeddings)

list(parser.model.parameters()) = [
    embeddings,                # (词表大小, 50)
    embed_to_hidden_weight,    # (1800, 200)
    embed_to_hidden_bias,      # (200,)
    hidden_to_logits_weight,   # (200, 3)

```

优化器拿到这个列表后，会追踪每个参数的梯度并持续更新它们。**普通张量（不是 `nn.Parameter`）不会出现在这个列表里，也不会被训练。**

**③ CrossEntropyLoss 内部做了什么？**

```python
loss_func = nn.CrossEntropyLoss()
# reduction='mean' (默认): 对batch取平均

loss = loss_func(logits, train_y)
# logits: (batch_size, 3)  ← 原始分数，未经过softmax
# train_y: (batch_size,)   ← 正确答案的类别索引 (0或1或2)
```

内部流程：

```python
# 伪代码：CrossEntropyLoss的内部实现
def cross_entropy_loss(logits, targets):
    # Step 1: log-softmax（数值稳定版）
    log_probs = log_softmax(logits)  # (batch, 3)
    
    # Step 2: 取出正确答案对应的log概率
    nll = -log_probs[range(batch), targets]  # (batch,)
    
    # Step 3: 取平均
    return nll.mean()
```

**为什么 `CrossEntropyLoss` 接收类别索引而不是 one-hot？**

这是 PyTorch 的设计选择——节省内存 + 更快。如果你的 `train_y` 是 one-hot 格式 `[[1,0,0], [0,1,0], ...]`，需要先转换：

```python
train_y = train_y.nonzero()[1]  # one-hot → 类别索引
```

**④ 为什么只保存验证集上最好的模型？**

```
训练集上的UAS：持续上升 ↗（模型一直在练训练数据）
验证集上的UAS：先升 ↗ 后降 ↘（模型开始"背答案"了）

                    ┌── 过拟合开始
          ↗↗↗↗↗↗↗↗↗↘↘↘↘↘↘↘
         ↗                  ↘
    ↗↗↗                        ↘↘↘
──────────────────────────────────────→ epoch
                ↑
         保存这个点的模型！
```

`torch.save(model.state_dict(), output_path)` 保存的是模型的**参数值**（所有权重和偏置的数值），不是整个模型对象。评估时用 `model.load_state_dict(torch.load(path))` 恢复。

### 6.3 `train_for_epoch` — 一轮训练（Part I）

```python
def train_for_epoch(parser, train_data, dev_data, optimizer, loss_func, batch_size):
    parser.model.train()      # 开启训练模式（Dropout生效）
    n_minibatches = math.ceil(len(train_data) / batch_size)
    loss_meter = AverageMeter()

    with tqdm(total=(n_minibatches)) as prog:
        for i, (train_x, train_y) in enumerate(minibatches(train_data, batch_size)):
            optimizer.zero_grad()                                      # ① 清空梯度
            
            train_x = torch.from_numpy(train_x).long()                 # 转Tensor
            train_y = torch.from_numpy(train_y.nonzero()[1]).long()    # one-hot→索引
            
            logits = parser.model(train_x)                             # ② 前向传播
            loss = loss_func(logits, train_y)                          # ③ 计算损失
            loss.backward()                                            # ④ 反向传播
            optimizer.step()                                           # ⑤ 更新参数
            
            prog.update(1)
            loss_meter.update(loss.item())

    print("Average Train Loss: {}".format(loss_meter.avg))
    print("Evaluating on dev set",)
    parser.model.eval()          # 关闭训练模式（Dropout不生效）
    dev_UAS, _ = parser.parse(dev_data)  # 在验证集上评估
    print("- dev UAS: {:.2f}".format(dev_UAS * 100.0))
    return dev_UAS
```

#### 代码要点

**① 训练的四步舞**

```python
# 这是所有PyTorch训练循环的标准模式：

optimizer.zero_grad()          # Step 1: 清空上一次迭代的梯度

logits = model(inputs)         # Step 2: 前向传播
loss = loss_func(logits, targets)

loss.backward()                # Step 3: 反向传播（计算所有参数的梯度）

optimizer.step()               # Step 4: 按照梯度更新参数
```

**② 为什么每次都要 `zero_grad()`？**

这是 PyTorch 新手最容易犯的错误！PyTorch 的梯度默认是**累加**的。

```python
# 如果不zero_grad()：
# 第1个batch: grad = [0.1, 0.2, 0.3]
# 第2个batch: grad = [0.1, 0.2, 0.3] + [0.15, 0.25, 0.35] = [0.25, 0.45, 0.65]  ← 错误！
# 第3个batch: grad = [0.25, 0.45, 0.65] + [...] = 越来越大 ← 完全错误！

# 正确做法：
# 每个batch之前清零
```

设计成累加而不是覆盖的原因：有些高级用法需要累加梯度（如做梯度累积来模拟更大的 batch size）。但对初学者来说，**每次迭代前 `zero_grad()` 是必写的固定步骤。**

**③ `loss.backward()` 自动计算了什么？**

当你写 `loss.backward()` 时，PyTorch 的 autograd 引擎做了以下事情：

```
loss (标量, 如 0.357)
  │
  ├→ ∂loss/∂logits = ...      (通过CrossEntropyLoss的导数)
  │     │
  │     ├→ ∂loss/∂W2 = ∂loss/∂logits × ∂logits/∂W2
  │     │   (存储到 hidden_to_logits_weight.grad)
  │     │
  │     ├→ ∂loss/∂b2 = ∂loss/∂logits × ∂logits/∂b2
  │     │   (存储到 hidden_to_logits_bias.grad)
  │     │
  │     └→ ∂loss/∂x_hidden (继续往前传)
  │           │
  │           ├→ ∂loss/∂W1 = ∂loss/∂x_hidden × ∂x_hidden/∂W1
  │           │   (存储到 embed_to_hidden_weight.grad)
  │           │
  │           ├→ ∂loss/∂b1 = ... 
  │           │   (存储到 embed_to_hidden_bias.grad)
  │           │
  │           └→ ∂loss/∂x_embed (继续往前传)
  │                 │
  │                 └→ ∂loss/∂embeddings = ...
  │                     (存储到 embeddings.grad)
```

这一切都是 PyTorch 在你写前向代码时"录制"下来的。每条你写的 `torch.matmul`、`F.relu`、`+` 操作，autograd 都知道对应的反向传播公式。你不需要手写任何导数。

**④ `optimizer.step()` 做了什么？**

拿到所有参数的 `.grad` 后，Adam 按照它的智能算法更新每个参数：

```python
# Adam更新（简化伪代码）
for param in model.parameters():
    # 1. 更新一阶矩（梯度方向的指数移动平均）
    m = beta1 * m + (1 - beta1) * param.grad
    
    # 2. 更新二阶矩（梯度大小的指数移动平均）
    v = beta2 * v + (1 - beta2) * param.grad**2
    
    # 3. 偏差修正（前几步的m,v倾向于0，需要修正）
    m_hat = m / (1 - beta1**t)
    v_hat = v / (1 - beta2**t)
    
    # 4. 更新参数
    param -= lr * m_hat / (sqrt(v_hat) + epsilon)
```

其中 `beta1=0.9`, `beta2=0.999`, `epsilon=1e-8` 是 Adam 的默认超参数。

**⑤ `train_y.nonzero()[1]` 的作用？**

```python
# train_y 的原始格式是 one-hot 编码：
train_y = [[1, 0, 0],    # S
           [0, 1, 0],    # LA
           [0, 0, 1],    # RA
           [1, 0, 0]]    # S

# CrossEntropyLoss 需要类别索引格式：
# train_y = [0, 1, 2, 0]

train_y.nonzero()   # 返回所有非零元素的 (行, 列) 坐标
# = (tensor([0, 1, 2, 3]), tensor([0, 1, 2, 0]))
#    ↑ 行索引                ↑ 列索引

train_y.nonzero()[1]  # 只取列索引 → 就是类别编号
# = tensor([0, 1, 2, 0])
```

因为 one-hot 中每一行只有一个 1，所以 `nonzero()[1]` 就是该行 1 所在的列号（0=S, 1=LA, 2=RA）。

**⑥ `parser.model(train_x)` 怎么触发 `forward` 的？**

```python
logits = parser.model(train_x)
# 不是 parser.model.forward(train_x) ← 虽然也能用但不推荐
```

这是 PyTorch `nn.Module` 的特殊机制：当你对一个 Module 实例做函数调用时，PyTorch 在内部调用 `__call__`，`__call__` 会做一些准备工作（如注册前向钩子等）然后调用 `forward`。

```python
# 等价关系：
output = model(x)       # 推荐
output = model.forward(x) # 能用但不推荐（跳过了 __call__ 的准备工作）
```

**⑦ `loss.item()` 是什么？**

```python
loss = loss_func(logits, train_y)  # loss 是一个 tensor(0.357)
loss.item()                         # 提取为 Python float: 0.357
```

`.item()` 把只含一个元素的张量提取为 Python 标量。用于打印和记录日志，不需要梯度追踪。

**⑧ `tqdm` 是什么？**

```python
with tqdm(total=n_minibatches) as prog:
    ...
    prog.update(1)  # 进度条+1
```

`tqdm` 是一个进度条库，在终端里显示训练进度：

```
76%|█████████████████████     | 38/50 [00:15<00:05, 2.34it/s]
```

训练神经网络通常要几分钟到几小时，没有进度条就像在黑屋子里走路——你不知道还要走多久。

## 8. 核心概念速查表

### 8.1 依存句法分析

| 概念 | 一句话解释 |
|------|-----------|
| **依存句法分析** | 找出句子中每个词依附于哪个词（head） |
| **Head（中心词）** | 被依赖的那个词，是关系的"上级" |
| **Dependent（依存词）** | 依赖别人的那个词，是关系的"下级" |
| **ROOT** | 人造的根节点，代表句子起点。所有句子都有一条从ROOT指向根词的弧 |
| **UAS** | Unlabeled Attachment Score，预测对 head 的比例。只看 head，不看关系类型 |

### 8.2 转换系统

| 概念 | 一句话解释 |
|------|-----------|
| **Stack（栈）** | 存放正在处理的词，LIFO。初始含ROOT |
| **Buffer（缓冲区）** | 存放还没处理的词，FIFO。初始含句子所有词 |
| **SHIFT (S)** | 从buffer移一个词到stack |
| **LEFT-ARC (LA)** | stack顶是head，前一个是dependent。箭头←。移除dependent |
| **RIGHT-ARC (RA)** | 前一个是head，stack顶是dependent。箭头→。移除dependent |
| **终止条件** | buffer空 + stack只剩["ROOT"] |
| **Oracle** | 标准答案生成器，告诉模型每一步的正确答案是什么 |

### 8.3 神经网络

| 概念 | 一句话解释 |
|------|-----------|
| **Embedding** | 把单词变成数字向量，"词的数学表示" |
| **nn.Parameter** | 给张量贴"我要被训练"的标签 |
| **Xavier初始化** | 让层间信号方差保持一致，防爆炸/消失 |
| **ReLU** | max(0,x)，负数清零，引入非线性 |
| **Dropout** | 训练时随机关神经元，防死记硬背 |
| **Softmax** | 把分数变成概率（所有输出加起来=1） |
| **CrossEntropyLoss** | 衡量预测概率分布和正确答案的差距 |
| **Logits** | 未经过softmax的原始输出分数 |

### 8.4 训练

| 概念 | 一句话解释 |
|------|-----------|
| **Epoch** | 把全部训练数据看一遍 |
| **Batch** | 一次处理的一小批样本（1024个） |
| **Adam** | 自适应学习率优化器，NLP默认选择 |
| **前向传播** | 输入→网络→输出 |
| **反向传播** | 从loss往回算每个参数的梯度（PyTorch自动完成） |
| **zero_grad()** | 清空累加的梯度，每次batch前必调 |
| **optimizer.step()** | 用梯度更新参数 |
| **model.train()** | 开启训练模式（Dropout生效） |
| **model.eval()** | 开启评估模式（Dropout关闭） |
| **过拟合** | 模型背下了训练数据，但新数据上表现差 |

### 8.5 PyTorch 关键 API

| API | 用途 | 在哪个文件用到 |
|-----|------|:---:|

| `optim.Adam` | Adam优化器 | run.py |
| `nn.CrossEntropyLoss` | 交叉熵损失 | run.py |
